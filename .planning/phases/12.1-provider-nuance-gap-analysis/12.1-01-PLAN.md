# PennyLLM Provider Gap Report

**Date:** 2026-03-17
**Covers:** 6 active providers (Cerebras, Google AI Studio, Groq, SambaNova, NVIDIA NIM, Mistral)
**Context:** PennyLLM v1.0 uses reactive 429-driven routing (Phase 12). This report audits each provider against the implemented abstractions.

## Executive Summary

- PennyLLM's reactive design (try -> 429 -> cooldown -> next in chain) handles most provider nuances correctly without pre-configured limits
- 6 providers analyzed, 0 have blocking gaps for v1 functionality
- 5 P0 documentation gaps must be addressed before v1 release
- 2 P1 code patches are low-effort improvements
- 4 P2 items deferred to Phase 13+ (credit tracking, per-model cooldown, pool quotas, per-second windows)

## Methodology

Three-category severity framework:
- **(a) Works but suboptimal** -- reactive handling produces correct behavior, but configuration is wasteful or a request is wasted
- **(b) Documentation/DX gaps** -- system works, but missing docs could lead to user confusion or misconfiguration
- **(c) Deferred to v2** -- genuine capability gaps requiring code changes beyond v1 scope

## Provider-by-Provider Analysis

### 1. Cerebras

**Limit Model Match:**
- Reality: Account-level limits (30 RPM, 60K TPM, 1M TPD). NOT per-key.
- PennyLLM: Per-key tracking. Mismatch but harmless -- reactive 429 catches it.

**Key Rotation Value:** NONE. All keys share account quota. Multiple keys waste config space.

**Error Response Handling:** Standard 429 with seconds-based rate limit headers. `parseRetryAfter()` and `parseRateLimitReset()` in `error-classifier.ts` handle correctly.

**Missing Capabilities:** None significant.

**DX Recommendations:**
- Document: "Only one Cerebras key needed per account -- multiple keys share the same limits"
- Warn in config validation if >1 Cerebras key detected (P1 candidate)

**Severity:** (a) works-but-suboptimal for multi-key config, (b) docs gap for key rotation guidance

**Confidence:** HIGH

### 2. Google AI Studio

**Limit Model Match:**
- Reality: Per-project limits (not per-key, not per-account). Per-model sub-limits within each project (e.g., Gemini 2.5 Flash = 10 RPM, Flash Lite = 15 RPM).
- PennyLLM: Per-key tracking. Keys from DIFFERENT projects get independent quota (strong use case). Keys from SAME project share quota (no benefit).

**Key Rotation Value:** HIGH -- but ONLY with keys from separate Google Cloud projects. Users can create up to 10 projects, each with independent quota. This is PennyLLM's strongest key rotation scenario.

**Per-Model Limits:** Models within a project have different RPM/TPM/RPD. PennyLLM cannot distinguish -- a key exhausted for one model might still work for another. Reactive 429 handles it (cools down, chain advances), but wastes a request.

**0/0/0 Models:** Some models have zero free-tier allocation. PennyLLM tries, gets 429, cools down. Works but wastes a request.

**Error Response Handling:** 429 with `x-ratelimit-reset` header. Parsed correctly by `parseRateLimitReset()`.

**DX Recommendations:**
- Document: "Create separate Google Cloud projects for each key to multiply quota"
- Document: "Keys from the same project share limits -- adding multiple same-project keys provides no benefit"
- List which models have 0/0/0 allocation (if discoverable) to help users avoid them

**Severity:** (a) per-model limits suboptimal, (b) CRITICAL docs gap for multi-project setup

**Confidence:** HIGH for structure, MEDIUM for exact numbers (Google cut quotas Dec 2025)

### 3. Groq

**Limit Model Match:**
- Reality: Per-organization, per-model. Four dimensions: RPM, RPD, TPM, TPD per model. Wildly different per-model (llama-3.1-8b: 14,400 RPD vs llama-3.3-70b: 1,000 RPD).
- PennyLLM: Per-key tracking. Same gap as Google -- exhausted for one model, still valid for others.

**Key Rotation Value:** NONE for same organization. All keys within one org share per-model quotas.

**TPM Constraint:** 6K-30K TPM is very tight. A single large prompt can exhaust per-minute token budget. Reactive 429 handles this well (cooldown, chain advances).

**Error Response Handling:** 429 with `x-ratelimit-reset-requests` header (epoch seconds). Parsed correctly.

**DX Recommendations:**
- Document: "Only one Groq key needed per organization"
- Document: "Keep prompts small for Groq -- TPM limits are tight (6K-30K depending on model)"
- Document: "Different Groq models have very different limits -- smaller models have much higher quotas"

**Severity:** (a) per-model limits suboptimal, (b) docs gap for TPM constraints and single-key guidance

**Confidence:** HIGH for production models, LOW for preview models

### 4. SambaNova

**Limit Model Match:**
- Reality: Per-model limits, two drastically different tiers:
  - Free tier: 20 RPM, 20 RPD, 200K TPD per model (almost unusable)
  - Developer tier (card linked, $0 balance): up to 1,440 RPM, 288K RPD per model (600x more)
- PennyLLM: Per-key tracking. No tier awareness.

**Key Rotation Value:** UNKNOWN (likely per-account, no benefit).

**The Critical Gap:** 600x difference between free and developer tier is unprecedented. Users who don't link a card get 20 RPD -- exhausted within minutes.

**$5 Signup Credit:** 30-day expiry. PennyLLM handles 402 as permanent session cooldown (correct for v1). Credit balance tracking deferred to Phase 13.

**Community SDK:** Uses `sambanova-ai-provider` (not first-party). Maintenance concerns noted.

**Error Response Handling:** 429 with epoch-based `x-ratelimit-reset-requests` header. Parsed correctly.

**DX Recommendations:**
- Document: "STRONGLY recommend linking a payment card (Developer tier at $0 balance). Free tier has only 20 requests per day per model -- practically unusable."
- Document: "$5 signup credit expires in 30 days"
- Flag as reactive-only for limit pre-configuration

**Severity:** (b) CRITICAL docs gap for tier recommendation, (c) credit tracking deferred to Phase 13

**Confidence:** HIGH for developer tier, MEDIUM for free tier (community-reported)

### 5. NVIDIA NIM

**Limit Model Match:**
- Reality: Rate-limited perpetual trial. ~40 RPM (community-reported). Per-model limits exist but NVIDIA explicitly refuses to publish.
- PennyLLM: Cannot pre-configure limits. 100% reactive -- correct approach.

**Key Rotation Value:** UNKNOWN (likely per-account, no benefit).

**Unpublished Limits:** NVIDIA stated they won't publish per-model limits. PennyLLM's reactive design is ideal here.

**Rate Limit Headers:** NOT documented by NVIDIA. LiteLLM maps `x-ratelimit-limit-requests` and `x-ratelimit-limit-tokens` but unconfirmed. If absent, PennyLLM defaults to 60s cooldown with exponential backoff -- correct behavior.

**402 Handling:** Legacy credit system produced 402. Credits reportedly removed, but 402 may still fire. PennyLLM handles 402 as permanent session cooldown -- correct.

**Geo-Restriction:** build.nvidia.com returns 403 from some countries. Cannot test locally from user's location.

**DX Recommendations:**
- Document: "NVIDIA limits are unpublished -- PennyLLM adapts automatically via 429 detection"
- Document: "Default cooldown timing may be conservative (60s) if NVIDIA doesn't return retry-after headers"
- Document: "build.nvidia.com may be geo-restricted in some regions"

**Severity:** (a) potentially conservative cooldown if no headers, (b) docs gap for reactive-only explanation

**Confidence:** LOW for exact limits, HIGH for reactive approach correctness

### 6. Mistral

**Limit Model Match:**
- Reality: Per-organization. 1 RPS, 50K TPM, 1B tokens/month. Possible pool-based shared quotas across model groups.
- PennyLLM: Has no `per-second` time window type. Cannot pre-model 1 RPS limit. Reactive 429 handles it.

**Key Rotation Value:** NONE. Up to 10 keys per account, all share same org limits.

**RPS Gap:** PennyLLM's `timeWindowSchema` supports per-minute, hourly, daily, monthly, rolling-30d -- but NOT per-second. Workaround: configure as ~2 RPM (conservative). Reactive 429 catches actual limit regardless.

**Pool System:** Community reports suggest models share token quotas in pools (general, coding, multimodal). PennyLLM has no pool concept. Reactive handling works.

**5-Minute Sliding Window:** Some large models reportedly use 5-minute windows. PennyLLM's `per-minute` can't model this. Reactive handling works.

**Data Privacy:** Free tier defaults to data sharing enabled. Opt-out available in admin console. CRITICAL documentation item.

**Error Response Handling:** Standard 429 with `Retry-After` header. Parsed correctly.

**DX Recommendations:**
- Document: "CRITICAL: Mistral free tier may use your data for training. Opt out in admin console before using."
- Document: "Only one Mistral key needed per account"
- Document: "Mistral uses per-second rate limiting (1 RPS) which PennyLLM handles reactively"

**Severity:** (b) CRITICAL docs for data privacy, (a) RPS gap but reactive handles it, (c) per-second window and pool system deferred to v2

**Confidence:** LOW for rate limits (community-reported), HIGH for data privacy concern

### 7. GitHub Models (Dropped)

Dropped from PennyLLM registry during Phase 12 execution. Registry comment: "doesn't offer meaningful free-tier value." Notes preserved in `docs/providers/notes/github-models.md` for reference. No gap analysis performed.

## Cross-Cutting Gaps

### Per-Model Limits (Google, Groq, SambaNova)

PennyLLM models limits at per-key level, but three providers enforce per-model limits within each key/account.

- A key exhausted for Model A might still have quota for Model B
- Chain executor skips entire provider after all keys get 429'd for one model
- Chain would come back to provider with a different model, but cooldown timer must expire first

**Impact:** Suboptimal utilization. Reactive approach still produces correct routing (no errors reach user), but quota is underutilized.
**Recommendation:** Document as known limitation. Phase 13+ could add per-model cooldown tracking.

### Account-Level Limits (Cerebras, Groq, Mistral)

Three providers have account/org-level limits where multiple keys share same quota pool.

**Impact:** Wasted configuration. Users might add multiple keys expecting quota multiplication.
**Recommendation:** Document clearly which providers benefit from key rotation vs don't:
- YES (different projects): Google AI Studio
- NO (account-level): Cerebras, Groq, Mistral, SambaNova (likely), NVIDIA NIM (likely)

### Per-Second Rate Limits (Mistral)

PennyLLM has no `per-second` time window type.

**Impact:** LOW -- reactive 429 handles it. User can't pre-configure Mistral's 1 RPS.
**Recommendation:** Add `'per-second'` to `timeWindowSchema` in a v1 patch (low effort).

### Unpublished/Unverified Limits (NVIDIA NIM, Mistral)

Cannot pre-configure limits with confidence. Reactive approach is correct strategy.

**Impact:** Users cannot use pre-configured limits for these providers. Must rely on reactive 429.
**Recommendation:** Flag as "reactive-only" in docs. Don't ship default limit policies for them.

### Missing Rate Limit Headers (NVIDIA NIM potentially)

If provider doesn't return `Retry-After` or reset headers, PennyLLM defaults to 60s cooldown.

**Impact:** Potentially conservative cooldown (waiting 60s when 5s would suffice).
**Recommendation:** Document. Empirical testing needed (blocked by geo-restriction).

## Phase 13+ Input
