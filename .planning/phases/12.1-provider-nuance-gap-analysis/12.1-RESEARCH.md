# Phase 12.1: Provider Nuance Gap Analysis - Research

**Researched:** 2026-03-17
**Domain:** Provider behavior vs PennyLLM abstraction mismatch analysis
**Confidence:** HIGH

## Summary

This phase is a pure analysis/documentation phase -- no code changes. The goal is to systematically compare each of PennyLLM's 6 target providers' actual behavior against the abstractions already implemented in Phase 12, categorize gaps by severity, and produce a prioritized gap report that feeds into Phase 13+ planning.

The reactive 429-driven routing design (implemented in Phase 12) makes many limit-modeling gaps less critical than they would be in a proactive system. The chain executor walks the model chain, tries the top-available model, and if it gets a 429, triggers cooldown and advances. This means gaps in pre-configured limit accuracy don't break routing -- they just waste a request. The primary remaining gaps are in developer experience (documentation, configuration guidance) and a few capability gaps that genuinely affect behavior.

**Primary recommendation:** Produce a structured gap report document with provider-by-provider analysis, categorized as (a) works-but-suboptimal, (b) documentation/DX gaps, (c) deferred-to-v2. The report should be actionable enough that Phase 13 planners can directly translate gap items into tasks.

## Architecture Patterns

### Gap Analysis Framework

The analysis uses a three-category severity framework already defined in the phase description:

**(a) Works but suboptimal** -- reactive 429 handling produces correct behavior, but configuration is wasteful or a request is wasted.

**(b) Documentation/DX gaps** -- the system works, but users need guidance to configure providers optimally. Missing docs could lead to confusion or misconfiguration.

**(c) Deferred to v2 (Phase 13+)** -- genuine capability gaps that require code changes beyond Phase 12's scope. These feed into the v2 roadmap.

### Provider-by-Provider Analysis Structure

Each provider section should cover:

1. **Limit model match** -- does PennyLLM's per-key limit model match reality?
2. **Key rotation value** -- does multi-key config genuinely multiply quota?
3. **Error response format** -- are 429/402 responses handled correctly?
4. **Missing capabilities** -- what can't PennyLLM model?
5. **DX recommendations** -- what should docs say?

## Provider Gap Analysis

### 1. Cerebras

**Limit model:** Account-level, not per-key. 30 RPM, 60K TPM, 1M TPD.
**Key rotation value:** NONE. Multiple keys all share the same account quota. Key rotation wastes config space.
**Error handling:** Standard 429 with rate limit headers (seconds-based reset). PennyLLM's `parseRetryAfter()` and `parseRateLimitReset()` handle this correctly.
**Category:** (a) Works but suboptimal -- multiple keys would all get 429'd simultaneously. Reactive handling works (all keys cool down, chain advances). Docs should note: "only one key needed per Cerebras account."
**Confidence:** HIGH -- well-documented limits, verified headers.

### 2. Google AI Studio

**Limit model:** Per-project (not per-key, not per-account). Three dimensions: RPM, TPM, RPD. Per-model limits within each project.
**Key rotation value:** HIGH -- but only if keys are from different projects. Keys within the same project share quota. Users can create up to 10 projects per account, each with independent quota. This is PennyLLM's strongest use case.
**Per-model limits:** Each model has different RPM/TPM/RPD (e.g., Gemini 2.5 Flash = 10 RPM vs Gemini 2.5 Flash Lite = 15 RPM). PennyLLM models limits per-key, not per-model. This means a key might still have quota for one model but be exhausted for another.
**0/0/0 models:** Some models listed in the dashboard have zero limits (not available on free tier). PennyLLM would try, get 429, and cooldown -- works but wastes a request.
**Error handling:** Google returns 429 with `x-ratelimit-reset` header. Already parsed by `parseRateLimitReset()`.
**Category:** (a) Per-model limits work suboptimally (wasted requests when one model is exhausted but others aren't). (b) Docs must emphasize: "use keys from DIFFERENT Google Cloud projects for true quota multiplication."
**Confidence:** HIGH for limit structure. MEDIUM for exact per-model numbers (Google cut quotas in Dec 2025, could change again).

### 3. Groq

**Limit model:** Per-organization, per-model. Four dimensions: RPM, RPD, TPM, TPD per model. Organization-level ceiling acts as cap.
**Key rotation value:** NONE for same organization. Multiple keys within one org share the same per-model quotas.
**Per-model limits:** This is the most granular provider. Each model has wildly different limits (e.g., llama-3.1-8b: 14,400 RPD vs llama-3.3-70b: 1,000 RPD). PennyLLM's per-key limit model cannot capture this -- a key exhausted for llama-3.3-70b might still have capacity for llama-3.1-8b.
**TPM constraint:** 6K-30K TPM is very tight. A single large prompt can exhaust the per-minute token budget. This is a binding constraint that PennyLLM's reactive approach handles well (429 triggers cooldown, chain advances).
**Error handling:** Groq returns 429 with `x-ratelimit-reset-requests` header (epoch seconds). Already parsed by `parseRateLimitReset()`.
**Category:** (a) Per-model limits mean suboptimal utilization when one model hits limits. (b) Docs should note tight TPM constraints and recommend keeping prompts small for Groq.
**Confidence:** HIGH for production models. LOW for preview models (may disappear).

### 4. SambaNova

**Limit model:** Per-model, two drastically different tiers. Free tier: 20 RPM, 20 RPD, 200K TPD per model. Developer tier (card linked, $0 balance): up to 1,440 RPM, 288K RPD per model.
**Key rotation value:** UNKNOWN -- limit scoping (per-key vs per-account) is not explicitly documented. Likely per-account based on similar providers.
**The critical gap:** 600x difference between free and developer tier. 20 RPD on free tier is almost unusable. Users who don't link a card will have a terrible experience.
**$5 signup credit:** 30-day expiry. PennyLLM's tier system supports `'trial'` with credits, but Phase 12 only does reactive 402 handling (permanent session cooldown). No credit balance tracking.
**Community SDK:** Uses `sambanova-ai-provider` (not first-party @ai-sdk/\*). May have stability/maintenance concerns.
**Error handling:** Returns 429 with epoch-based `x-ratelimit-reset-requests` header. Parsed correctly by existing code.
**Category:** (b) CRITICAL docs gap -- must strongly recommend Developer tier (link card). (a) Per-model limits same gap as Groq/Google. (c) Credit depletion tracking deferred to Phase 13.
**Confidence:** HIGH for developer tier limits (verified from docs). MEDIUM for free tier (20 RPD is community-reported).

### 5. NVIDIA NIM

**Limit model:** Rate-limited perpetual trial. ~40 RPM (community-reported). Per-model limits exist but are explicitly unpublished by NVIDIA.
**Key rotation value:** UNKNOWN. Limits are likely per-account (developer program account).
**Unpublished limits:** NVIDIA explicitly stated they won't publish per-model limits. PennyLLM must rely 100% on reactive 429 detection. Cannot pre-configure limits with any confidence.
**Rate limit headers:** NOT documented by NVIDIA. LiteLLM integration suggests `x-ratelimit-limit-requests` and `x-ratelimit-limit-tokens` may be present, but unconfirmed. If no `Retry-After` or rate limit reset headers exist, PennyLLM falls back to default cooldown (60s) with exponential backoff -- this is correct behavior.
**402 handling:** Legacy credit system produced 402. NVIDIA says credits are gone, but 402 may still fire for edge cases. PennyLLM already handles 402 as permanent session cooldown -- correct.
**Geo-restriction:** build.nvidia.com returns 403 from user's country. Cannot test locally.
**Category:** (a) Unpublished limits mean PennyLLM can only react, never pre-configure -- this is fine for reactive design. (b) Docs should note: "NVIDIA limits are unpublished, PennyLLM adapts automatically via 429 detection." (c) No rate limit headers may mean suboptimal cooldown duration (defaults to 60s backoff instead of precise retry-after).
**Confidence:** LOW for exact limits (unpublished). HIGH for reactive handling approach (it's the only approach possible).

### 6. Mistral

**Limit model:** Per-organization (not per-key). 1 RPS, 50K TPM, 1B tokens/month. Possible pool-based shared quotas across model groups.
**Key rotation value:** NONE. Up to 10 keys per account, all share same limits.
**RPS vs RPM gap:** Mistral uses requests-per-second (1 RPS), not requests-per-minute. PennyLLM's `timeWindowSchema` supports `'per-minute'`, `'hourly'`, `'daily'`, `'monthly'`, `'rolling-30d'` -- but NOT `'per-second'`. This is a genuine capability gap. Workaround: configure as ~2 RPM (conservative approximation), but reactive 429 handling will catch the actual limit regardless.
**Pool system:** Community reports suggest models share token quotas in pools (general, coding, multimodal). PennyLLM has no concept of shared pools. Reactive handling works -- hitting TPM on one model triggers cooldown, chain advances to different provider.
**5-minute sliding window:** Some large models reportedly use 5-minute sliding windows for token tracking. PennyLLM's `per-minute` window can't model this. Again, reactive handling works.
**Data privacy:** Free tier may use data for training by default. Opt-out available in admin console. This is a CRITICAL documentation item.
**Error handling:** Standard 429 with `Retry-After` header. Parsed correctly.
**Category:** (b) CRITICAL docs gap -- data privacy opt-out. (a) RPS constraint can't be pre-modeled but reactive 429 catches it. (c) Per-second window type deferred to v2 (low priority given reactive design). (b) Docs should note single-key-only and pool sharing.
**Confidence:** LOW for rate limits (not publicly documented, community-reported). HIGH for data privacy concern.

## Cross-Cutting Gap Summary

### Gap: Per-Model Limits (affects Google, Groq, SambaNova)

PennyLLM models limits at the per-key level, but Google, Groq, and SambaNova enforce per-model limits within each key. This means:

- A key exhausted for Model A might still have quota for Model B
- PennyLLM's chain executor skips the entire provider after all keys get 429'd for one model
- The chain would eventually come back to this provider with a different model (if one exists in the chain), but the cooldown timer must expire first

**Impact:** Suboptimal utilization. The reactive approach still produces correct routing behavior (no errors reach the user), but quota is underutilized.
**Recommendation:** Document as known limitation. Phase 13+ could add per-model cooldown tracking (track which model caused the 429, not just which key/provider).

### Gap: Account-Level vs Per-Key Limits (affects Cerebras, Groq, Mistral)

Three providers have account/organization-level limits where multiple keys share the same quota pool. Key rotation provides zero benefit.
**Impact:** Wasted configuration. Users might add multiple keys expecting quota multiplication.
**Recommendation:** Docs should clearly state which providers benefit from key rotation (Google: YES via separate projects) and which don't (Cerebras, Groq, Mistral: NO).

### Gap: Per-Second Rate Limits (affects Mistral)

PennyLLM has no `per-second` time window type. Mistral's 1 RPS limit cannot be pre-modeled.
**Impact:** LOW -- reactive 429 handling catches the actual limit. The user just can't pre-configure it.
**Recommendation:** Add `'per-second'` to the `timeWindowSchema` enum and `DURATIONS` in a future phase. For now, document the workaround.

### Gap: Unpublished/Unverified Limits (affects NVIDIA NIM, Mistral)

NVIDIA explicitly refuses to publish limits. Mistral's limits are behind authentication and community-reported.
**Impact:** Cannot pre-configure limits with confidence. PennyLLM's reactive approach is the correct strategy here.
**Recommendation:** In the gap report, flag these providers as "reactive-only" -- don't ship default limit policies for them.

### Gap: Missing Rate Limit Headers (potentially affects NVIDIA NIM)

If NVIDIA doesn't return `Retry-After` or `x-ratelimit-reset-*` headers, PennyLLM falls back to default cooldown (60s) with exponential backoff. This may be longer than necessary.
**Impact:** Suboptimal cooldown duration (potentially waiting 60s when 5s would suffice).
**Recommendation:** Document. Testing needed to discover actual headers (blocked by geo-restriction for this user).

## Don't Hand-Roll

| Problem                  | Don't Build                    | Use Instead                        | Why                                                                                    |
| ------------------------ | ------------------------------ | ---------------------------------- | -------------------------------------------------------------------------------------- |
| Per-model limit tracking | Custom per-model cooldown map  | Reactive 429 per-provider cooldown | Matches reactive design, per-model tracking adds complexity for marginal benefit in v1 |
| Per-second rate limiting | Custom sub-second throttle     | Reactive 429 handling              | Provider enforces it anyway; pre-modeling adds no value                                |
| Credit balance tracking  | Custom credit deduction system | Reactive 402 detection             | Phase 13 scope; 402 permanent cooldown is sufficient for v1                            |

## Common Pitfalls

### Pitfall 1: Assuming Key Rotation Always Helps

**What goes wrong:** User configures 5 Cerebras keys expecting 5x quota, gets exactly the same limits.
**Why it happens:** Account-level limits mean all keys share quota.
**How to avoid:** Documentation must clearly state per-provider key rotation value.
**Warning signs:** Multiple keys for same provider all hitting 429 simultaneously.

### Pitfall 2: SambaNova Free Tier Unusability

**What goes wrong:** User configures SambaNova without linking a card, gets 20 RPD, provider exhausted within minutes.
**Why it happens:** 600x difference between free and developer tier is unprecedented.
**How to avoid:** Documentation must prominently recommend linking a card ($0 balance works).
**Warning signs:** SambaNova always first to exhaust in chain.

### Pitfall 3: Mistral Data Privacy Default

**What goes wrong:** User's prompts used for Mistral model training without knowledge.
**Why it happens:** Free tier defaults to data sharing enabled.
**How to avoid:** Documentation must include privacy opt-out instructions before any Mistral setup.
**Warning signs:** N/A (silent, no technical symptom).

### Pitfall 4: Confusing Per-Model and Per-Key Limits

**What goes wrong:** User thinks provider is fully exhausted when only one model hit its limit.
**Why it happens:** PennyLLM cooldown is per-provider (after all keys tried), not per-model.
**How to avoid:** Document this limitation. Users can work around by placing multiple models from the same provider at different chain positions.
**Warning signs:** Provider cooldown fires but user knows they have quota for a different model.

### Pitfall 5: Google Keys from Same Project

**What goes wrong:** User adds 5 Google keys from the same project expecting 5x quota, gets 1x.
**Why it happens:** Per-project enforcement means same-project keys share quota.
**How to avoid:** Documentation must explain: "create separate Google Cloud projects for each key."
**Warning signs:** All Google keys 429 simultaneously with same retry-after values.
