**1. Create `src/policy/PolicyEngine.ts`**:

The PolicyEngine class holds resolved policies, a storage reference, and EventEmitter. It provides the `evaluate()` method that checks usage against limits.

Constructor:

```typescript
constructor(
  resolvedPolicies: ResolvedPolicy[],
  storage: StorageBackend,
  emitter: EventEmitter,
  options?: { warningThreshold?: number }
)
```

- Store `resolvedPolicies` in a Map keyed by `${provider}:${keyIndex}` for O(1) lookup
- Store `storage`, `emitter`, `warningThreshold` (default 0.8)
- Initialize `firedWarnings: Set<string>` for event deduplication
- Import debug with namespace `llm-router:policy:engine`

`evaluate()` method:

```typescript
async evaluate(
  provider: string,
  keyIndex: number,
  estimatedTokens?: { prompt: number; completion: number }
): Promise<EvaluationResult>
```

- Look up ResolvedPolicy by `${provider}:${keyIndex}` key
- If no policy found (custom provider with no limits), return `{ eligible: true, limits: [], enforcement: 'hard-block' }`
- If policy has empty limits array, return `{ eligible: true, limits: [], enforcement: policy.enforcement }`
- Query `storage.getUsage()` for ALL limits concurrently using `Promise.all()`
- For each limit, build `LimitStatus`:
  - `current`: usage value from storage. For limits with `type` of `tokens`, `daily`, or `monthly` — this is the total tokens from getUsage(). For `calls` — this is the call count (getUsage returns totalTokens, but call-based limits need a different approach. **Important**: `getUsage()` returns a single number (totalTokens). For call-based limits, storage tracks calls separately. Since StorageBackend.getUsage currently returns total tokens, call-based and rate-based limits will need to store their values via the same mechanism but with different window semantics. For now, the evaluator treats the returned value as the counter for whatever limit type is being checked — the caller (Phase 4 usage tracking) is responsible for incrementing the correct counter.)
  - `max`: limit.value
  - `remaining`: `Math.max(0, limit.value - current)`
  - `percentUsed`: `(current / limit.value) * 100`
  - `resetAt`: Calculate from `limit.window` — `new Date(Math.ceil(Date.now() / limit.window.durationMs) * limit.window.durationMs)` (next window boundary)
- If `estimatedTokens` provided, add estimated usage to current values for token-based limits before checking eligibility (pre-call check). This means: for token-type limits, compare `current + estimatedTokens.prompt + estimatedTokens.completion` against max, not just `current`.
- Determine `eligible`: all limits must have `current < max` (or `current + estimated < max` if estimated provided)
- Determine `closestLimit`: the limit with highest `percentUsed`
- **Event emission** (after building results):
  - For each limit where `percentUsed >= warningThreshold * 100`: check if `${provider}:${keyIndex}:${limit.type}` is in `firedWarnings` Set. If not, emit `limit:warning` event matching `LimitWarningEvent` shape and add to Set.
  - For each limit where `percentUsed >= 100`: emit `limit:exceeded` event matching `LimitExceededEvent` shape (no deduplication on exceeded — fire every time to ensure visibility)
- Return `EvaluationResult`

Additional methods:

- `getPolicy(provider: string, keyIndex: number): ResolvedPolicy | undefined` — expose resolved policy for inspection
- `getAllPolicies(): ResolvedPolicy[]` — return all resolved policies (for router.getConfig())
- `resetWarnings(): void` — clear firedWarnings Set (useful for testing or when usage drops)

**2. Create `src/policy/staleness.ts`**:

`checkStaleness(resolvedPolicies: ResolvedPolicy[], emitter: EventEmitter): void`

- Import debug with namespace `llm-router:policy:staleness`
- Iterate over resolved policies
- Skip policies without `metadata?.researchedDate` (user-configured limits have no staleness concept)
- For policies with `researchedDate`: calculate days since research
- If >30 days old:
  - Log via debug: `Stale policy detected: ${provider} (researched ${researchedDate})`
  - Emit `policy:stale` event with `PolicyStaleEvent` shape: `{ provider, researchedDate, daysOld, suggestion }` where suggestion is `"Run 'npm update llm-router' or verify limits at ${sourceUrl}"` (fall back to generic message if no sourceUrl)
- **Deduplicate by provider** — only fire once per provider even if multiple keys share the same shipped policy
