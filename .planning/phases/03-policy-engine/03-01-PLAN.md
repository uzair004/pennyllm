  <name>Task 2: Implement policy resolver with three-layer merge</name>

**1. Create `src/policy/resolver.ts`** with:

- `mergeLimits(...layers: PolicyLimit[][]): PolicyLimit[]` — Merge limit arrays where later layers override earlier ones. Match by composite key `${type}:${window.type}`. Later layer replaces earlier for same key. Unmatched limits from any layer are included (additive for new types, replacement for matching types).

- `resolvePolicies(config: RouterConfig, shippedDefaults: Map<string, Policy>): ResolvedPolicy[]` — Core resolver function:
  1. Iterate over `config.providers` entries
  2. Skip providers where `enabled === false`
  3. For each provider, look up shipped default from `shippedDefaults` map
  4. **Duplicate key detection**: Collect all key strings (from string keys and object keys), throw ConfigError if duplicates found within same provider. Error message: `Duplicate key detected in provider "${provider}": keys must be unique`
  5. For each key in `provider.keys`:
     - Extract key string: `typeof key === 'string' ? key : key.key`
     - Extract per-key limits: `typeof key === 'string' ? undefined : key.limits`
     - Three-layer merge: `mergeLimits(shipped?.limits ?? [], provider.limits ?? [], perKeyLimits ?? [])`
     - If `perKeyLimits` is `[]` (empty array), treat as "no per-key overrides" (same as omitting) — empty array does NOT clear inherited limits
     - Enforcement: carry over from shipped default if available, otherwise default to `'hard-block'`
     - Metadata: carry over from shipped default if available (for staleness checks)
     - Create `ResolvedPolicy` with provider, keyId (the key string), keyIndex (position in array), merged limits, enforcement, metadata
  6. **Custom providers** (no shipped default): If a provider has no shipped default AND no configured limits (neither provider-level nor per-key), create ResolvedPolicy with empty limits array. Log debug warning: `No shipped policy or configured limits for provider "${provider}" — keys will always be considered available`
  7. **Validate contradictory limits**: After merging, for each limit type that appears in multiple windows, check that narrower windows have values <= wider windows of the same type. E.g., if both `tokens:daily` and `tokens:monthly` exist, daily value must be <= monthly value. Throw ConfigError if violated: `Contradictory limits for provider "${provider}": daily ${type} limit (${daily}) exceeds monthly ${type} limit (${monthly})`
     - Only compare: daily <= monthly, hourly <= daily, per-minute <= hourly (for same limit type)
     - Do NOT compare across different limit types (tokens vs calls vs rate)

- Import `debug` with namespace `llm-router:policy:resolver`
