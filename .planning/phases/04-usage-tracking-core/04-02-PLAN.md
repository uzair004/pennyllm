  <files>src/usage/UsageTracker.ts, src/usage/index.ts</files>
  <action>
Create the core UsageTracker class in `src/usage/UsageTracker.ts`:

**Constructor:**

- Accepts: `storage: StorageBackend`, `policies: ResolvedPolicy[]`, `emitter: EventEmitter`, `estimationConfig: EstimationConfig`
- Stores references to storage, emitter, estimationConfig
- Creates internal `CooldownManager` instance
- Creates internal `Set<string>` for requestId deduplication (`recordedRequests`)
- Creates internal `Map<string, number>` for estimatedRecords count per key (`${provider}:${keyIndex}`)
- Builds a `Map<string, ResolvedPolicy>` keyed by `${provider}:${keyIndex}` for O(1) policy lookup (same pattern as PolicyEngine)
- Use `debug('llm-router:usage')` for all debug logging

**estimate() method:**

- Signature: `estimate(messages: Array<{ role: string; content: unknown }>, options?: { system?: string; tools?: unknown[]; maxTokens?: number }): EstimationResult | null`
- Delegates to `estimateTokens()` from estimation.ts, passing `this.estimationConfig`
- Returns `EstimationResult | null`

**record() method:**

- Signature: `async record(provider: string, keyIndex: number, usage: { promptTokens: number; completionTokens: number } | null, requestId: string, estimation: EstimationResult | null): Promise<void>`
- **Fire-and-forget pattern**: This method is called but its Promise is not awaited by the caller. All errors are caught and debug-logged, never thrown.
- Wrap entire body in try-catch. On error, debug-log and return.
- **Deduplication**: Check if `recordedRequests` Set has `requestId`. If yes, debug-log skip and return. Add requestId to Set after recording. Clean up Set entries lazily: if Set size > 10000, clear it (prevents unbounded growth; exact dedup is best-effort since fire-and-forget means the window for duplicates is tiny).
- **Determine tokens**: If `usage` is non-null, use it (provider-reported). If null, fall back to `estimation` (if non-null, treat as `{ promptTokens: estimation.prompt, completionTokens: estimation.completion }`). If both null, use `{ promptTokens: 0, completionTokens: 0 }`.
- **Determine estimated flag**: `estimated = !usage` (true when provider didn't report usage).
- **Track estimatedRecords**: If estimated, increment `estimatedRecords` Map for `${provider}:${keyIndex}`.
- **Multi-window recording**: Look up the ResolvedPolicy for `provider:keyIndex`. For each limit in the policy's limits array, call `this.storage.increment(provider, keyIndex, { prompt: tokens.promptTokens, completion: tokens.completionTokens }, limit.window, 1)`. The `1` is the callCount (every recording is exactly one call). If no policy found, skip recording (no windows to record to).
- **Rolling 30-day special handling**: For `rolling-30d` window type limits, the increment already goes to a daily bucket (because getPeriodKey returns daily key). No special handling needed here.
- **Emit usage:recorded event**: After all increments complete, emit `usage:recorded` event with payload: `{ timestamp: Date.now(), requestId, provider, keyIndex, promptTokens: tokens.promptTokens, completionTokens: tokens.completionTokens, estimated, windows: policy.limits.map(l => l.window.type) }`.

**handle429() method:**

- Signature: `handle429(provider: string, keyIndex: number, retryAfterHeader?: string): void`
- Delegates to `this.cooldown.setCooldown(provider, keyIndex, retryAfterHeader)`
- Also calls `record429Call()` internally: increment call count for all applicable windows WITHOUT incrementing tokens. For each limit in the policy, call `storage.increment(provider, keyIndex, { prompt: 0, completion: 0 }, limit.window, 1)`. This counts the 429 as a call per CONTEXT.md decision. Wrap in try-catch, debug-log errors.

**getUsage() method:**

- Overloads: `getUsage(): Promise<UsageSnapshot>` and `getUsage(provider: string): Promise<ProviderUsage>`
- Implementation: `async getUsage(provider?: string): Promise<UsageSnapshot | ProviderUsage>`
- If `provider` specified, build `ProviderUsage` for that provider only. Otherwise build `UsageSnapshot` for all providers.
- For each provider/keyIndex in the policies Map:
  - For each limit in the policy: query `storage.getUsage(provider, keyIndex, limit.window)` to get structured usage. For rolling-30d: query last 30 daily buckets by calling `storage.getUsage()` with daily window and adjusted timestamps. Sum the 30 results. For other window types: single query.
  - Build `KeyUsageWindow` for each limit with: type, promptTokens, completionTokens, totalTokens, callCount, limit (from policy limit.value), remaining (limit.value - relevant usage metric based on limit.type), percentUsed, resetAt (from `getResetAt()`).
  - Remaining calculation: for `tokens` type limits, use totalTokens. For `calls` type limits, use callCount. For `rate` type limits, use callCount. For `daily`/`monthly` type limits, use totalTokens.
  - Get cooldown status from `this.cooldown.getCooldown(provider, keyIndex)`.
  - Get estimatedRecords count from the internal Map.
  - Assemble `KeyUsage`.
- Sum totals across all keys for `ProviderUsage.totals`.
- Return `UsageSnapshot` with ISO timestamp or `ProviderUsage`.

**resetUsage() method:**

- Signature: `async resetUsage(provider?: string, keyIndex?: number): Promise<void>`
- Call `storage.resetAll(provider, keyIndex)` to clear storage.
- If no args: also clear cooldowns (`cooldown.clearAll()`), requestId set, estimatedRecords map.
- If provider only: clear cooldowns for all keys of that provider, clear estimatedRecords for that provider.
- If both: clear cooldown for that specific key, clear estimatedRecords for that key.

**getCooldownManager() method:**

- Returns internal CooldownManager (for Phase 5 selection to check cooldown status).
