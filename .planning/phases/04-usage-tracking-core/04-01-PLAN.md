Create the `src/usage/` directory with 5 files:

**src/usage/types.ts** - Usage API types per CONTEXT.md getUsage() shape:

- `EstimationConfig` interface: `{ defaultMaxTokens: number; tokenEstimator?: (text: string) => number }`
- `EstimationResult` interface: `{ prompt: number; completion: number }`
- `KeyUsageWindow` interface: `{ type: TimeWindow['type']; promptTokens: number; completionTokens: number; totalTokens: number; callCount: number; limit: number; remaining: number; percentUsed: number; resetAt: string }` (all primitive/string values per JSON-serializable requirement)
- `KeyUsage` interface: `{ keyIndex: number; windows: KeyUsageWindow[]; cooldown: { active: boolean; until: string | null; reason: string | null }; estimatedRecords: number }`
- `ProviderUsage` interface: `{ provider: string; keys: KeyUsage[]; totals: { promptTokens: number; completionTokens: number; totalTokens: number; callCount: number } }`
- `UsageSnapshot` interface: `{ providers: ProviderUsage[]; timestamp: string }`
- `StructuredUsage` interface: `{ promptTokens: number; completionTokens: number; totalTokens: number; callCount: number }` (return type for updated StorageBackend.getUsage)

**src/usage/periods.ts** - Period calculation per CONTEXT.md window period decisions:

- `getPeriodKey(window: TimeWindow, timestamp: number): string` function handling all 5 window types:
  - `per-minute`: `Math.floor(timestamp / 60000).toString()` (fixed 60-second boundaries, uses durationMs from window)
  - `hourly`: ISO slice to `YYYY-MM-DDTHH` (UTC hour boundaries)
  - `daily`: ISO slice to `YYYY-MM-DD` (UTC day boundaries)
  - `monthly`: ISO slice to `YYYY-MM` (calendar month, NOT duration division)
  - `rolling-30d`: Same as daily (daily bucket for summation)
- `getResetAt(window: TimeWindow, timestamp: number): Date` function:
  - `per-minute`: Next 60-second boundary from timestamp
  - `hourly`: Next UTC hour boundary
  - `daily`: Next UTC midnight
  - `monthly`: First of next month at UTC midnight
  - `rolling-30d`: Next UTC midnight (daily bucket boundary)
- Use `new Date(timestamp)` and `.toISOString()` for calendar-based keys. Use `Date.UTC()` for computing next boundaries.

**src/usage/estimation.ts** - Token estimation per CONTEXT.md:

- `estimateTokens(messages: Array<{ role: string; content: unknown }>, options: { system?: string; tools?: unknown[]; maxTokens?: number }, config: EstimationConfig): EstimationResult | null`
- Wrap entire function body in try-catch. On any error, debug-log via `debug('llm-router:usage')` and return `null`.
- For prompt estimation: concatenate all text content from messages (skip non-string content for image/file inputs), add system prompt text if present, add JSON.stringify of tools array if present. Use `config.tokenEstimator` if provided, otherwise `defaultCharRatioEstimator` (~4 chars/token via `Math.ceil(text.length / 4)`).
- For completion estimation: use `options.maxTokens` if set, otherwise `config.defaultMaxTokens`.
- Export `defaultCharRatioEstimator` separately for reuse.

**src/usage/cooldown.ts** - Cooldown manager per CONTEXT.md 429 decisions:

- `CooldownManager` class with private `Map<string, { until: number; reason: string }>` where key is `provider:keyIndex`.
- `setCooldown(provider, keyIndex, retryAfterHeader?, defaultCooldownMs = 60_000)`: Parse Retry-After header - try parseInt first (seconds), if NaN try Date constructor (HTTP date), if both fail use defaultCooldownMs. Store `{ until: Date.now() + cooldownMs, reason: '429 rate limit' }`.
- `isInCooldown(provider, keyIndex)`: Check if cooldown exists and not expired. Delete expired entries (lazy cleanup). Returns boolean.
- `getCooldown(provider, keyIndex)`: Returns `{ until: string; reason: string } | null` where `until` is ISO string. Returns null if expired or not found.
- `clearAll()`: Clear all cooldowns (for resetUsage).
- `clear(provider, keyIndex)`: Clear specific cooldown.

**src/usage/index.ts** - Barrel exports:

- Export all types from `./types.js`
- Export `getPeriodKey`, `getResetAt` from `./periods.js`
- Export `estimateTokens`, `defaultCharRatioEstimator` from `./estimation.js`
- Export `CooldownManager` from `./cooldown.js`

**src/types/interfaces.ts** - Extend StorageBackend interface:

- Update `increment()` signature to add optional `callCount?: number` parameter (defaults to 0). This is a BACKWARD-COMPATIBLE change - existing callers (PolicyEngine) don't pass callCount and get 0.
- Update `getUsage()` return type from `Promise<number>` to `Promise<{ promptTokens: number; completionTokens: number; totalTokens: number; callCount: number }>`. This is a BREAKING change to the interface. PolicyEngine.evaluate() currently calls `getUsage()` and expects a `number`. It will need a compatibility fix (see below).
- Add `resetAll(provider?: string, keyIndex?: number): Promise<void>` method for bulk reset operations.

**src/storage/MemoryStorage.ts** - Update implementation:

- Import `getPeriodKey` from `../usage/periods.js`.
- Replace `makeKey()` to use `getPeriodKey(window, Date.now())` instead of `Math.floor(Date.now() / window.durationMs)`. New format: `${provider}:${keyIndex}:${window.type}:${getPeriodKey(window, Date.now())}`.
- Update `increment()` to accept `callCount?: number` parameter. Track callCount in UsageRecord. Since UsageRecord doesn't have a `callCount` field, store it in a parallel `Map<string, number>` called `callCounts` that maps the same composite key to a call count integer. Increment by `callCount ?? 0` on each call.
- Update `getUsage()` to return `{ promptTokens, completionTokens, totalTokens, callCount }` instead of just `number`. Read from both the main data Map (for tokens) and the callCounts Map.
- Update `cleanupExpired()` to also clean up the callCounts Map entries for expired periods. To detect expired entries without duration-only division, compare period keys: for calendar-based windows (hourly, daily, monthly, rolling-30d), parse the stored period key string vs current period key. For per-minute, keep the existing `Math.floor(timestamp / durationMs) < currentPeriod` approach. Simplify: iterate data Map entries, extract the period portion from the composite key (4th segment after splitting on `:`), compare with `getPeriodKey(record.window, Date.now())`. If they differ, the record is expired.
- Implement `resetAll(provider?, keyIndex?)`: If no args, clear all data + callCounts. If provider only, delete entries starting with `${provider}:`. If both, delete entries starting with `${provider}:${keyIndex}:`.
- Update `reset()` to also clear the corresponding callCounts entry.
- Update `close()` to also clear callCounts Map.

**Fix PolicyEngine compatibility**: PolicyEngine.evaluate() in `src/policy/PolicyEngine.ts` calls `this.storage.getUsage()` and expects a number. Update it to destructure the new return: `const { totalTokens: current } = await this.storage.getUsage(...)`. This is a one-line change per call site (there is one call in the limitStatusPromises map).
